# ADR-0023 — Codex documentation layer committed to the repository

**Date:** 2026-05-13
**Status:** Accepted

---

## Context

Spellbook has a Codex layer: plain-language explanations for every `Public/` script (`.codex/Public/*.md`) and session summaries for each development day (`.codex/sessions/YYYY-MM-DD.md`). These are generated by the Codex hook system — PostToolUse fires on every write and updates the relevant explanation file; a retroactive generator (`retro.py`) backfills history from git.

The question is whether `.codex/` should be committed to the repository or excluded via `.gitignore`.

Generated output is typically gitignored: build artefacts, compiled binaries, coverage reports. The default assumption is that generated files are reproducible on demand and add noise to diffs.

---

## Decision

Commit `.codex/` to the repository and keep it public.

---

## Reasoning

The Codex files are not build artefacts. They are curated teaching documents that require the Anthropic API, the full git history, and a working Codex installation to reproduce. A developer cloning the repo cold cannot regenerate them without setting up the entire toolchain.

More importantly, they serve a different audience than the source code. The `.ps1` scripts are for engineers running commands. The `.codex/` explanations are for engineers reading and learning the codebase — including future contributors, the co-engineer, and anyone studying the repo as a reference. Hiding them behind a `.gitignore` removes that value from GitHub entirely.

The session summaries in `.codex/sessions/` also function as a development log — a human-readable record of what was built each day and why. This is distinct from git log messages, which record *what changed* but not *what was being built or decided*.

The PostToolUse hook keeps the file explanations current automatically. The session summaries require a manual `retro.py` run at session end, but that cost is low and the output is durable.

---

## Consequences

- Anyone viewing the GitHub repository can read plain-language explanations of every command alongside the source.
- Diffs will include `.codex/` updates whenever a `Public/` script changes. These diffs are meaningful — they show the explanation evolving alongside the code.
- The repository carries a dependency on the Codex toolchain for `.codex/` to stay current. Engineers without Codex installed will still have the committed explanations; they just won't auto-update on their local writes.
- `retro.py` is available in the Codex repository (`Big-Bronson/Codex`) for regenerating the full layer against any project.

---

## Alternatives considered

**Gitignore `.codex/`** — Keeps the repo lighter and avoids explanation drift if someone edits a script without Codex installed. Rejected because the explanations have independent value as documentation and cannot be trivially regenerated by a cold reader.

**Separate documentation repository** — Moves the explanation layer out of Spellbook entirely. Rejected as too much overhead for a two-person project; keeping docs co-located with code is the pattern the ADR system already follows.
